Global alert is in most cases initiated by the application or system, but can also be used when an important message needs to be flagged. It’s a general message to all users and are not associated with user actions. Global alert appears on all pages of the application/website and extends in full length above the header. It can include links or actions such as expand and close.

NB: Remember to make the global alert fixed to the top of your application and make sure that it extends the full length of the page!

Global alerts can also have a close button. Global alerts can also be expandable. Add and remove the clas e-alert--open to open and close the alert. Global alerts can also have actions. Local alerts are messages related to the site and the content. It can be placed anywhere between the header and footer, but close to the content it is related to. Local messages can also be used for feedback after a user actions. The maximum width is 850px. The status of an alert helps to communicate the information and the severity of the alert. Each status consists of a defined color and an icon. It is important to use the correct status of the alert to provide a consistent and universal user experience.
error e-alert--error
Notify when something goes wrong or the user is denied completing their task. Explain why and what to do. Example: Error, missing fields and invalid entry. This status can be used with local and global alerts.
warning e-alert--warn
Help users to avoid error situations and to give attention to something undesirable that may occur. Explain why and what to do. This status can be used with local and global alerts.
informative e-alert--info
Inform users to understand minor contextual changes, and are not typically in response to a user action. Like a helpful tip or a suggestion. This status can be used with all alerts.
When new alerts appear, users of assistive technologies may miss them. ARIA live regions enable programmatically exposing new alerts in a way that can be announced by assistive technology. Only alerts that require the user's immediate attention should use role=alert . When this role is applied to an element, the browser sends an accessible alert event to assistive technology products, which subsequently notifies the user.

Alerts will only be announced when the content of the element with role="alert" is updated. Do not try to dynamically add/generate an alert with role="alert" that is already populated with the alert message you want to be announced – this generally does not lead to an announcement, as it is not a content change.

Instead, you can use a visually hidden alert container for screen reader notifications. Using JavaScript, you can dynamically change the content inside the element with role="alert" . We advise replacing the entire alert message at once instead of just parts of the alert message to ensure it fires an update to assistive technologies.

If the content inside the element with role="alert" is initially hidden using CSS, making it visible will also cause the alert to fire. However, ensure that the container is not hidden using display:none , as this will hide it from assistive technologies, meaning that users won't be notified of any alerts.

Use role=status when an alert contains advisory information that is not important enough to be an alert. Use role="note" when a alerts whose subject matter is supplementary to the primary subject.
