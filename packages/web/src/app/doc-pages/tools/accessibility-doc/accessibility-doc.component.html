Accessibility is often linked to disabilities, but designing for permanent disabilities also benefits those with temporary or situational limitations. For example, solutions for the deaf can help users in noisy environments. Inclusive design improves experiences for everyone.

Web Content Accessibility Guidelines (WCAG) ensure web accessibility.

  • WCAG 2.0 (2008) became an ISO standard in 2012.
  • WCAG 2.1 (2018) was adopted as a requirement in 2020 and legally applied in 2021. It builds on WCAG 2.0, adding better support for mobile devices.
  • Applies to intranets, extranets, and mobile apps.

Read the full guidelines on the w3c official site or see a summary of WCAG 2.1 on the Norwegian Digitization Directorate .

Accessible Rich Internet Applications (ARIA) are roles and attributes that should be added to web content to make the content more accessible. When creating code it is important to always check if the content should have any aria attributes or roles depending on the purpose of the content.

  • Alternative text: Use alt="" for images; leave empty for decorative ones.
  • ARIA labels: Provide text alternatives for non-visible elements.
  • ARIA hidden: Hide elements from assistive tools when necessary.

The markup of HTML using tags is one of the most important ways to ensure your content is accessible. Use the correct HTML tags like <h1>, <h2>, <p>, <nav>, <section>, <article> etc. for your elements as much as possible throughout your code. It is also important to note the order of your elements, <h1> before <h2> and so on.

Here are some tips to achieve the desired keyboard navigation in the code. Read more about keyboard navigation in general further down on this page.

  • Group elements together to make navigation more efficient and easy.
  • Using radio buttons or any other HTML element as base elements can sometimes be handy to get the wanted logic inherently.
  • Tabindex can be used to make an element focusable, by setting the tabindex="0" or a positive number. To make an already focusable element non-focusable, apply tabindex="-1".
  • All form elements can use the .e-form-field classes for label, error messages and alignment of form elements (see Text Field ).
  • The class .e-sr-only can be used to remove the element visually while still available for screen readers.
  • Use background classes .e-color-background-* to automatically get the correct text contrast.

Test with screen readers and tools like Axe and SonarCloud, but prioritize real-user testing.

Write accessible digital content:

  • Use concise, meaningful titles.
  • Add alt text for images.
  • Provide subtitles for videos.
  • Use descriptive link text instead of “click here.”

Keyboard navigation should take care of people who can’t access the website with a mouse. This can be people who are blind, partially sighted, or impaired motor skills in the arms and hands. A user should be able to use the tab key to navigate and activate every intersactive element on a page.

  • Tab – navigate forward to the next interactive element
  • Shift+Tab – navigate backwards to the previous interactive element
  • Space – activate the interactive element. Check or uncheck a checkbox and toggle
  • Enter – activate the interactive element
  • Arrow keys - move within an element. Up/Down: move between radio buttons and options in a dropdown. Right/Left: adjust slider, move between dates in a date picker and options in radio filter, segmented control and tab
  • Esc – close a modal or dropdown and return focus to the element that activated it

It should be easy to see the focused element on the page. Our focus indicator is a blue outline around the interactive element with 2px padding. This blue color is not part of Elvia's identity color palette and should only be used for keyboard navigation. The reason for this is to let the focus color stand out even more from our content and profile.

Follow the visual order (left to right, top to bottom). Related controls should use a single tab stop with arrow key navigation.

  • Non-clickable elements should not be possible to reach, or elements that are temporarily hidden and not visible for somebody using a mouse
  • If you get a tooltip on hover with the mouse, you should get the same when you focus the element
  • Watch out for keyboard traps, the user should always be able to leave a dialog or go back
  • If an interaction reveals hidden content, ensure the focus is moved to the revealed content
