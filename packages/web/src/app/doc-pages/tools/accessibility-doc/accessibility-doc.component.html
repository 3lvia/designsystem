Accessibility
We create accessible solutions to enable all users to access an user experience, regardless of permanent, temporary or situational disability.
Traditionally, accessibility is often linked with people with disabilities, but when we design services and products for users with a permanent disabilities, someone with a temporary or situational limitation will benefit as well. For example, a solution that works well for someone who is deaf might also benefit any person in a loud environment. Designing inclusive solutions for a spectrum of disabilities connects different people in similar circumstances and makes it better for many.

Web Content Accessibility Guidelines (WCAG) are a collection of guidelines developed by the Web Accessibility Initiative for making web content more accessible.

WCAG 2.0 was published in December 2008 and became an ISO standard in October 2012. In 2018, version 2.1 was introduced as recommendations, but was as of July 2020 adopted as future requirements with an intended transition period of six months.

By January 2021, WCAG 2.1 will be applied as legal requirements. It is backwards-compatible with WCAG 2.0 with some additions that, amonst other, takes more consideration to mobile devices than previously. Note that the requirements apply to intranets, extranets, as well as mobile apps.

You can read the full set of guidelines on the w3c official site , or read a summary of the new additions to 2.1 on the Norwegian Digitization Directorate Norwegian Digitization Directorate .

Here’s a list of recommendations to make sure your code is accessible:
  • Use alt text for images
  • Check color contrasts
  • Test your code on screen readers
  • All form elements can use the e-form-field classes for label, error messages and alignment of form elements (see input )
  • Use background classes (e-bg) to automatically get right text contrasts

 What to think about when writing accessible in digital media:
  • Use concise and meaningful titles to make pages easier to scan
  • Use alt text for images, and image descriptions when suitable
  • Use subtitles in videos when possible
  • Use describing phrases rather than “click here” or “read more” for buttons and links

Keyboard accessibility is one of the most important aspects of web accessibility. A keyboard user typically uses the Tab key to navigate through interactive elements on a web page. Focus adds an additional outline around the element. Version 2.1 of the Web Content Accessibility Guidelines (WCAG) specify a minimal contrast ratio of 3:1 against adjacent colour(s) for the parts, which are required to understand the meaning or content of a graphical object.

The outline should not be visible on click. The correct behavior is included on our components when using CDN. If you are not using CDN you have to remember to add the elvis script. Read more on set up and getting started.

This outline comes with the elements that require focus state in elvis. If you want to add focus to your custom elements you can use the class e-outline .
